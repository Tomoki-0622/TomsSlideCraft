# 🎨 Tom's SlideCraft — 仕様書

> 📅 作成日: 2026-07-01

---

## 🗂️ 1. プロジェクト概要

社内向けプレゼン資料作成Webアプリ。AIとの対話でストーリーを設計し、マークダウン → Marpスライド → PPTX までワンストップで作成・編集できる。

---

## ✨ 2. 機能一覧

| # | 機能 | 概要 |
|---|---|---|
| 1️⃣ | ストーリー設計 | AIチャットでプレゼン構成を検討し、マークダウンファイルとしてエクスポート |
| 2️⃣ | スライド生成 | マークダウンをもとにMarpでHTML（パワポ風）スライドを生成。PPTXダウンロードも可能 |
| 3️⃣ | スライド編集 | AI指示チャットで自然言語修正 ＋ モード切替によるマークダウン直接編集。アイコン検索・挿入が容易 |

## 🛠️ 3. 技術スタック

| 項目 | 採用技術 | 備考 |
|---|---|---|
| 🏗️ フレームワーク | Next.js（App Router） | |
| 🎨 UIスタイリング | Tailwind CSS ＋ shadcn/ui | |
| 🤖 AIバックエンド | OpenAI API（GPT-4o） | サーバーサイド専用環境変数で管理 |
| 🖼️ スライド生成 | Marp（`@marp-team/marp-core`） | Node.js環境必須。Route Handlerに `export const runtime = 'nodejs'` を明示。またはクライアントサイドで `dynamic import + ssr:false` を使用 |
| 📊 PPTX出力 | PptxGenJS（テキスト・構成重視、デザイン簡略化） | タイムアウト対策のためクライアントサイドで実行 |

#### 操作フロー

- AIがマークダウン全体を返答したタイミングで自動更新（ストリーミング完了時）
- 生マークダウンテキストをシンタックスハイライト付きで表示（Marpレンダリングはしない）

#### 機能詳細

- **左パネルモード切替**: 「AIチャット」「マークダウン直接編集」をトグルで切り替え可能
- **AIへの修正指示チャット**: 自然言語で指示 → マークダウンを自動更新。アイコン検索で取得したURLとともに「〇枚目の右上に配置して」と指示することでスライドに反映可能
- **マークダウン直接編集モード**: テキストエディタでマークダウンを直接編集。変更はリアルタイムでプレビューに反映

- **HTML出力**: Marpが生成したHTMLをダウンロード
- **PPTXダウンロード**: PptxGenJSでテキスト・構成をそのままPPTX化（デザインは簡略化）
- **テーマ選択**: MVPでは `toms-slidecraft` のみ（ボタンは表示するが非活性）

## 🤖 7. AIストーリー設計の仕様

### System Prompt（ひな形）

```
あなたは社内プレゼン資料のストーリー設計を支援するAIアシスタントです。
以下のルールに従ってください：
1. まず5つのテンプレートパターンを提示し、ユーザーに選択を促す
2. 選択後、共通ヒアリング3項目を1問ずつ順番に質問する
3. ヒアリング完了後、選択パターンに沿ったスライド構成案を提示する
4. ユーザーの修正指示に従ってマークダウンを更新する
5. 出力するマークダウンは必ずMarpフロントマター付きの完全なファイルとする
6. スライドに絵文字は使用しない。チャット返答には絵文字を使ってよい
7. キーメッセージは必ず主語＋述語の完全文で書く（体言止め禁止）
```

### 会話履歴管理

- 保持ウィンドウ: 直近 **10往復**（超過した古いメッセージを切り捨て）
- マークダウン全体は毎回コンテキストに含めず、変更差分のみをAIに渡す
- トークン上限（128k）の80%超過を検知した場合、ユーザーに警告トーストを表示する

### 共通ヒアリング項目（全テンプレート共通）

1. プレゼンのタイトル・テーマ
2. 対象の聴衆（役職・部署・意思決定権限）
3. このプレゼンで得たいアクション（承認・理解・意見収集 等）

> パターン5（フリー入力）でも上記3項目は必ずヒアリングする（スキップ不可）。
> ヒアリング完了後、AIがゴールと目次の深掘りに移行する。

## 🔐 8. 認証仕様

- ログイン方式: パスワード共有型（全員同一パスワード）
- パスワード管理: Vercel環境変数（`SITE_PASSWORD`）。`NEXT_PUBLIC_` プレフィックス禁止
- セッション管理: httpOnly Cookie（ログイン後は再入力不要）

### Cookie 設定

```
Set-Cookie: session=<token>; HttpOnly; SameSite=Lax; Secure; Max-Age=604800
```

- `HttpOnly`: JavaScriptからのアクセス禁止（XSS対策）
- `SameSite=Lax`: CSRF攻撃の軽減
- `Secure`: HTTPS通信のみ送信
- `Max-Age=604800`: セッション有効期限 **7日間**

### ログアウト

- ヘッダー右端に「ログアウト」ボタンを配置
- クリックでCookieを削除しログイン画面にリダイレクト

### パスワード変更手順（運用）

1. Vercelダッシュボードで `SITE_PASSWORD` 環境変数を更新
2. Vercelで再デプロイを実行（自動デプロイまたは手動トリガー）
3. 全ユーザーに新パスワードを周知（既存Cookieは無効化されないため再ログインを促す）

> 退職者が発生した場合は速やかに上記手順でパスワードを変更すること。将来的には個人アカウント管理（§10）への移行を優先課題とする。

## 💾 9. データ保存

- 保存先: ブラウザ LocalStorage
- 保存内容: 作成済みマークダウン、プロジェクト名、最終更新日時
- 注意: ブラウザをまたいだ共有・復元は非対応（MVP版）

### LocalStorageキー設計

```
slidecraft:projects          // プロジェクトID一覧（JSON配列）
slidecraft:project:<uuid>    // 各プロジェクトのデータ（JSON）
```

各プロジェクトのデータ構造（詳細は §15 付録参照）：

```json
{
  "id": "<uuid>",
  "name": "CRM導入提案",
  "markdown": "---\nmarp: true\n...",
  "updatedAt": "2026-07-01T23:00:00Z"
}
```

### 容量制限対処

- LocalStorageの上限は通常 **5MB**
- 保存時にサイズをチェックし、4MB超過で警告トースト、5MB超過で保存をブロックしてエラーを表示
- エラー時は「不要なプロジェクトを削除してください」と案内する
- IndexedDBへのフォールバックは将来拡張候補（§10）

## 📐 11. スライド作成ルール

### 基本ルール

- **マークダウンの内容を忠実に反映する** — 勝手に内容を追加・削除しない
- **1セクション = 1スライド**が基本（内容が多い場合は2枚に分割してよい）

---

### チェックシート準拠ルール（必須）

以下は社内プレゼン作成チェックシートに基づく必須ルール。

#### 1. 目的を押さえる

- **ぼやかした表現を使わない**: 「〇〇を検討する」「〇〇を推進する」など主語・期限・責任者が不明な表現は禁止
- **次の行動が明確**: クロージングスライドで「誰が・いつまでに・何をするか」を明示する

#### 2. ストーリー・表現

- **全スライドにキーメッセージ/結論を必須配置**: スライドヘッダー直下に「このスライドで何が言いたいか」を1文で記載する（配置場所を全スライドで統一）
- **体言止め・主語なし名詞表現はNG**: キーメッセージは必ず主語＋述語の完全文で書く
  - NG例: 「データ品質の課題」
  - OK例: 「データ品質の低さがAI活用の最大の阻害要因である」
- **1スライド1メッセージ**: 1枚のスライドで伝えることは1つに絞る
- **大枠→細部の順**: いきなり細部から入らず、まず全体像を提示する
- **資料冒頭にサマリを記載**: タイトル直後にエグゼクティブサマリスライドを入れる

#### 3. デザイン

- **フォント**: 日本語は `游ゴシック`（既定）、英語は `Calibri` を優先
- **本文フォントサイズは `text-base`（16px）以上**: `text-sm`（14px）は注釈・補足のみ使用可
- **Suntoryロゴを全スライドのフッター右側に必須配置**: Marpカスタムテーマの CSS `section::after` で固定配置する（AIによる毎回挿入は誤削除リスクがあるため禁止）
  - 使用ファイル: `public/suntory-logo.png`（リポジトリに含める前に法務・広報部門の使用許諾を必ず確認すること）
  - 実装例: `styles/marp-theme.css` の `section::after { content: url('/suntory-logo.png'); position: absolute; bottom: 16px; right: 24px; width: 80px; }`
- **デザイン4原則を意識**:
  - **近接**: 意味のある塊をグルーピング、余白を有効活用
  - **整列**: 文字/オブジェクトの端を揃える（中央揃えとバラバラ配置は避ける）
  - **反復**: フォント・サイズ・色・スライド構成を全体で統一（スライドマスタ相当）
  - **コントラスト**: 強調したい要素にのみコントラストを使う
- **文字強調は色ではなく太さ・大きさで表現する**: `<strong>` タグを活用し、色による強調は避ける
- **ページ番号必須**: ヘッダー右端に `NN / MM` 形式で表示（既定テンプレートで実装済み）
- **スライドには絵文字を使用しない**: 絵文字はチャット返答（§12）にのみ使用する。スライドコンテンツでは代わりにアイコン画像（Iconify）を使う
- **文中に改行コードを挿入しない**: スライド幅に収まる文章に `<br>` などの明示的な改行は入れない（表示崩れの原因）

### スライド構成の設計

パターンに応じたスライド構成を生成する。必ず含めるスライド：

1. **タイトルスライド** — タイトル・日付・作成者の会社名、部署名、氏名（キーメッセージも1行で）
2. **アジェンダスライド** — 全セクションの目次
3. **各セクションスライド** — マークダウンのセクション順に生成
4. **まとめスライド** — ゴール・次のステップ（ある場合）

#### スライドの内容生成ルール

- セクションタイトルはそのままスライドタイトルに使う
- 箇条書きは `<ul>/<li>` または `<ul class="two-column">` で整形
- 重要ポイント（`note`フィールドや★マーク）は `<div class="highlight-box">` で強調

#### キーメッセージバーの生成（全セクションスライドに必須）

スライドヘッダーの直下に、必ずキーメッセージバーを挿入する。Marp内ではTailwindCSSは動作しないため、**インラインスタイルのみ**使用する：

```html
<!-- キーメッセージ（必須・体言止めNG・主語＋述語で1文） -->
<div style="padding: 8px 20px; background: #F5F5F5; border-bottom: 1px solid rgba(84,195,225,0.3); flex-shrink: 0;">
  <p style="font-size: 14px; font-weight: bold; color: #1a3a5c; line-height: 1.4; margin: 0;">
    ▌ {このスライドの結論を主語＋述語の完全文1文で。体言止め・ぼやかした表現は禁止}
  </p>
</div>
```

キーメッセージの書き方ルール：

| 種別 | 例 |
|---|---|
| NG（体言止め） | 「データ品質の課題」「AI活用の推進」 |
| NG（ぼやかし） | 「データ品質を検討する」「AI活用を推進する」 |
| OK | 「データ品質の低さが全社AI活用の最大の阻害要因である」 |
| OK | 「Option Aを採用し2027年Q4までに全社展開を完了する計画を提案する」 |

## 📝 12. マークダウン出力スタイルガイド（チャット返答専用）

> このセクションは **AIのチャット返答（ストーリー設計の会話メッセージ）** のスタイルガイドである。スライドコンテンツ（マークダウンのスライド本文）には適用しない。スライドのルールは §11 を参照。

AIがチャット返答として出力するテキストは、人間が読んで直感的に理解できるよう、以下の方針に従って作成する。

### ✅ 基本方針

- **絵文字を積極的に使う**: 見出し・箇条書きの先頭に絵文字を付けて視覚的に分かりやすくする
- **内容に合った絵文字を選ぶ**: 課題なら ⚠️、ゴールなら 🎯、コストなら 💰、スケジュールなら 📅 など
- **読み流せる構造にする**: 重要度に応じて `**太字**` や `> 引用` を使い、パッと見て要点が伝わるようにする
- **過剰な絵文字は避ける**: 1つの箇条書きに絵文字は1つまで。文中への多用はしない

### 🖋️ 絵文字の使い方例

```markdown
## 🎯 本日のゴール
- ✅ 投資可否の承認
- 📋 方針の合意
- ⚠️ 検討に戻すポイントの確認

## 💰 コスト試算
| 区分 | 金額 |
|---|---|
| 🏗️ 初期費用 | 500万円 |
| 🔄 年間運用費 | 120万円 |

## ⚠️ リスクと対策
- **リスク**: ベンダーロックイン
- **対策**: マルチクラウド構成を採用
```

## 🚨 13. エラーハンドリング仕様

### エラー種別と対処方針

| エラー種別 | トリガー | UI表示 | リトライ |
|---|---|---|---|
| OpenAI API 失敗（5xx） | チャット送信・マークダウン生成 | チャット内インラインエラー「AI応答に失敗しました。再送信してください」 | 手動再送信ボタン表示 |
| OpenAI API レート制限（429） | 同上 | 「しばらくお待ちください（約30秒後に再試行してください）」 | 30秒後に自動リトライ |
| ネットワークエラー | 全APIコール | トースト通知「ネットワーク接続を確認してください」 | 手動リトライボタン |
| LocalStorage 保存失敗（QuotaExceeded） | プロジェクト保存 | トースト「容量上限に達しました。不要なプロジェクトを削除してください」 | なし（削除後に再保存） |
| Marpレンダリングエラー | スライドプレビュー更新 | プレビューエリアに「レンダリングエラー: マークダウンの構文を確認してください」 | 自動（マークダウン編集で再トリガー） |
| Iconify API 障害 | アイコン検索 | 検索結果欄に「アイコン検索が利用できません。URLを直接入力してください」 | 手動リトライボタン |
| 認証エラー（Cookie失効） | 全画面 | ログイン画面にリダイレクト | 再ログイン |

## 🔒 14. セキュリティ・環境変数仕様

### 環境変数

| 変数名 | 種別 | 説明 |
|---|---|---|
| `SITE_PASSWORD` | サーバーサイド専用 | ログイン共有パスワード |
| `OPENAI_API_KEY` | サーバーサイド専用 | OpenAI APIキー |

- **`NEXT_PUBLIC_` プレフィックスは絶対に付けない** — クライアントバンドルに含まれてしまう
- どちらの変数もブラウザのDevToolsから参照できてはならない

### Content Security Policy（CSP）

Next.js の `next.config.js` に以下のCSPヘッダーを設定し、XSSリスクを軽減する：

```js
// next.config.js
headers: [
  {
    source: '/(.*)',
    headers: [
      { key: 'Content-Security-Policy',
        value: "default-src 'self'; script-src 'self'; img-src 'self' https://api.iconify.design data:; style-src 'self' 'unsafe-inline';" },
      { key: 'X-Frame-Options', value: 'DENY' },
      { key: 'X-Content-Type-Options', value: 'nosniff' },
    ],
  },
]
```

### LocalStorageのセキュリティリスク

- 社内機密情報がLocalStorageに平文保存されることを利用者に周知する
- CSPによりXSSリスクを軽減するが、完全な保護は不可能
- 機密度の高いプレゼン内容はブラウザ共有PCで作成しないよう運用ルールを設ける
