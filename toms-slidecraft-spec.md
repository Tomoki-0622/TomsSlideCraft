## 📋 0. 要件定義

### 0-1. ビジネス背景・課題

サントリーホールディングス株式会社 デジタル&AI本部 グローバルITG では、以下の課題を抱えている：

- **プレゼン資料の作成に時間がかかりすぎる** — ストーリー設計からスライド完成まで多大な工数を要している
- **英語化（翻訳）に時間がかかる** — 日本語で作成した資料を英語版に変換する作業が別途発生している

### 0-2. 解決目標（成功基準）

| 指標 | 目標値 |
|---|---|
| プレゼン作成時間の削減率 | **70%削減** |

### 0-3. ステークホルダー

| 役割 | 対象 |
|---|---|
| **エンドユーザー** | サントリーHD デジタル&AI本部 グローバルITG メンバー（約50名） |
| **システム管理者** | Tomさん（1名）— Vercelデプロイ・環境変数管理・パスワード変更を担当 |

### 0-4. スコープ

#### MVPに含めるもの
- AIチャットによるプレゼンストーリー設計
- Marpを使ったスライド生成・プレビュー・編集
- HTML / PPTX ダウンロード
- **日→英 全スライド翻訳（翻訳結果を新規プロジェクトとして保存）**
- **スクリプト作成（日本語・英語対訳、スライドサムネイル連動）**
- 共有パスワード認証
- ブラウザ LocalStorage によるプロジェクト保存

#### MVPに含めないもの（§10 参照）
- サーバーサイドDB・複数デバイス対応
- 個人アカウント管理
- チーム共有・コメント機能
- スライドデザインテーマの追加
- 画像アップロード

### 0-7. 前提条件

- 利用者は Google Chrome または Microsoft Edge の最新版を使用できる
- 利用者はインターネット接続環境を持つ
- Azure OpenAI のデプロイは MVPリリースまでに利用可能になる（デプロイ名は決定次第 `AZURE_OPENAI_DEPLOYMENT` 環境変数に設定）
- Suntoryロゴ画像（`public/suntory-logo.png`）の使用は法務・広報部門の許諾を得たものとする

### 0-8. データガバナンス方針

- Azure OpenAI（社内環境）を使用するため、プレゼン内容が外部に送信されるリスクは低い
- ただし、以下の注意事項をバナーで表示する：

> 「本ツールで入力した内容は社内Azure OpenAI環境に送信されます。社内情報取り扱いルールに従ってご利用ください。」

**バナーの表示タイミング**:
- LocalStorageに `slidecraft:governance_accepted` フラグが存在しない場合のみ、ログイン後のダッシュボード上部に表示する
- ユーザーが「確認しました」ボタンをクリックすると `slidecraft:governance_accepted: true` をLocalStorageに保存し、以降は表示しない（**生涯1回のみ表示**）
- ブラウザ・端末を変えるとLocalStorageがリセットされるため再表示される（許容範囲とする）

### ストーリー作成タブ

#### レイアウト

```
┌──────────────────────────┬───────────────────────┐
│                          │                       │
│   AIチャット              │  マークダウンプレビュー │
│   （会話形式）            │  （生マークダウンテキスト│
│   AIがパターンを提示      │   シンタックスハイライト）│
│   → 共通質問を1問ずつ     │                       │
│      ヒアリング           │  [マークダウンをエクスポート]│
│  [最初からやり直す]        │  [スライド編集で開く]  │
│  [入力欄] [送信]          │                       │
└──────────────────────────┴───────────────────────┘
```

#### 操作フロー

1. AIが§7に定義された5テンプレートを選択肢として提示 → ユーザーが番号で選択
2. AIが共通質問3項目を1問ずつヒアリング
3. 3問完了後、AIが構成案（スライドタイトル一覧）を提示 → フリーチャットで修正・深掘り
4. 「マークダウンをエクスポート」でファイルダウンロード、または「スライド編集で開く」でLocalStorage経由で資料の作成タブへ受け渡し
   - 資料の作成タブに既に別のプロジェクトが開いていた場合は確認ダイアログを表示
5. 「最初からやり直す」ボタン: 「チャット履歴とマークダウンをクリアします。LocalStorageのプロジェクトデータは残ります。」の確認ダイアログ後にチャット履歴とマークダウンプレビューをクリアし、パターン選択に戻る（LocalStorageのプロジェクトデータは削除しない）

#### 状態管理（sessionStorage）

- ストーリー設計中のチャット履歴とマークダウンは `sessionStorage` に自動保存する
  - `sessionStorage` はタブ・ブラウザを閉じると自動でクリアされる
  - 同一タブ内での誤リロード時は内容が復元される（ブラウザクラッシュ保護）
- アプリを開くたびに**常に白紙から開始**する（前回セッションのデータは復元しない）

#### AI生成中のUX

- 送信ボタンをローディングスピナーに変え、チャット欄に「生成中...」のプレースホルダーメッセージを表示
- AI応答が完了したタイミングで一括でチャット・マークダウンを更新（ストリーミングなし）

#### マークダウンプレビューの更新タイミング

- AI応答が完了し、マークダウン全体を受け取ったタイミングで一括更新（ストリーミングなし）
- 生マークダウンテキストをシンタックスハイライト付きで表示（Marpレンダリングはしない）

#### 機能詳細

- **左パネルモード切替**: 「AIチャット」「マークダウン直接編集」をトグルで切り替え可能
- **AIへの修正指示チャット**: 自然言語で指示 → マークダウンを自動更新。アイコン検索で取得したURLとともに「〇枚目の右上に配置して」と指示することでスライドに反映可能
  - **AIコンテキスト**: 指示送信時に**マークダウン全文 ＋ ユーザー指示**をAzure OpenAIに渡す（差分ではなく全文を毎回送ることで正確な編集位置の特定を保証）
  - **AI生成中のUX**: 送信ボタンをローディングスピナーに変え、チャット欄に「生成中...」のプレースホルダーメッセージを表示。生成完了後に一括でマークダウンを更新する（ストリーミングなし）
- **マークダウン直接編集モード**: テキストエディタでマークダウンを直接編集。変更はリアルタイムでプレビューに反映
- **保存**: 手動保存ボタン（`💾 保存`）をツールバーに配置
  - **未保存インジケーター**: ボタン横に「● 未保存」を表示。保存後は「✓ 保存済み」に切り替わる
  - ブラウザを閉じる際の未保存警告（`beforeunload`）はなし
- **アイコン検索パネル（左下・高さ全体の1/3）**: キーワード検索でIconify APIからSVGアイコンを一覧表示。「チャットに送る」ボタンでURLをチャット入力欄に自動挿入（コピー→貼り付けの手間を削減）
- **スライドナビ**: 左右矢印ボタン ＋ 下部サムネイル一覧
- **HTML出力**: Marpが生成したHTMLをダウンロード
- **PPTXダウンロード**: PptxGenJSでテキスト・構成をそのままPPTX化（デザインは簡略化）
- **テーマ選択**: MVPでは `toms-slidecraft` のみ（ボタンは表示するが非活性。ホバー時のツールチップ「MVP後に追加予定」を表示）
- **タブ切り替え時の未保存状態**: 未保存インジケーターが表示されていてもタブ切り替え警告は**表示しない**（`beforeunload` 警告も同様に出さない）
- **ダウンロードファイル名規則**: `{プロジェクト名}_YYYYMMDD.html` / `.pptx`（日付はダウンロード実行日）

## 🔐 8. 認証仕様

- ログイン方式: パスワード共有型（全員同一パスワード）
- パスワード管理: Vercel環境変数（`SITE_PASSWORD`）。`NEXT_PUBLIC_` プレフィックス禁止
- セッション管理: httpOnly Cookie（ログイン後は再入力不要）

### Cookie 設定

```
Set-Cookie: session=<token>; HttpOnly; SameSite=Lax; Secure; Max-Age=604800
```

- `HttpOnly`: JavaScriptからのアクセス禁止（XSS対策）
- `SameSite=Lax`: CSRF攻撃の軽減
- `Secure`: HTTPS通信のみ送信
- `Max-Age=604800`: セッション有効期限 **7日間**

### トークン生成・検証方式

- `<token>` は **`HMAC-SHA256(randomUUID + passwordHash, SECRET_KEY)`** で生成する
  - `passwordHash`: `SITE_PASSWORD` のSHA-256ハッシュ
  - `SECRET_KEY`: Vercel環境変数 `SESSION_SECRET`（別途追加）
- **パスワード変更時の自動無効化**: トークン検証時に現在の `passwordHash` で再検証するため、`SITE_PASSWORD` を変更・再デプロイすると既存の全Cookieが自動的に無効化される（再ログインが必要になる）
- これにより退職者のブラウザに残ったCookieはパスワード変更後に即時失効する

### ログアウト

- ヘッダー右端に「ログアウト」ボタンを配置
- クリックでCookieを削除しログイン画面にリダイレクト

### パスワード変更手順（運用）

1. Vercelダッシュボードで `SITE_PASSWORD` 環境変数を更新
2. Vercelで再デプロイを実行（自動デプロイまたは手動トリガー）
3. 全ユーザーに新パスワードを周知（既存Cookieは無効化されないため再ログインを促す）

> 退職者が発生した場合は速やかに上記手順でパスワードを変更すること。将来的には個人アカウント管理（§10）への移行を優先課題とする。

## 💾 9. データ保存

- 保存先: ブラウザ LocalStorage
- 保存内容: 作成済みマークダウン、プロジェクト名、最終更新日時
- 注意: ブラウザをまたいだ共有・復元は非対応（MVP版）

### LocalStorageキー設計

```
slidecraft:projects             // プロジェクトID一覧（JSON配列）
slidecraft:project:<uuid>       // 各プロジェクトのデータ（JSON）
slidecraft:script:jp:<uuid>     // 日本語スクリプトのキャッシュ（JSON）
slidecraft:script:en:<uuid>     // 英語スクリプトのキャッシュ（JSON）
slidecraft:governance_accepted  // ガバナンスバナー承認フラグ（boolean: true）
```

各プロジェクトのデータ構造（詳細は §16 付録参照）：

```json
{
  "id": "<uuid>",
  "name": "CRM導入提案",
  "markdown": "---\nmarp: true\n...",
  "updatedAt": "2026-07-01T23:00:00Z"
}
```

スクリプトキャッシュのデータ構造（日本語・英語それぞれ同じ構造。詳細は §16 付録参照）：

```json
{
  "projectId": "<uuid>",
  "sourceUpdatedAt": "2026-07-01T23:00:00Z",
  "script": "スライド 1:\n皆さん、おはようございます...\n\nスライド 2:\nまず背景として..."
}
```

### データ削除ルール

- プロジェクトを削除する際は以下のキーをすべて連動削除する：
  - `slidecraft:project:<uuid>`
  - `slidecraft:script:jp:<uuid>`
  - `slidecraft:script:en:<uuid>`
- `(EN)` プロジェクト（`parentProjectId` が設定されているプロジェクト）を親プロジェクトと一緒に削除するか個別に削除するかはダイアログで確認する

### 容量制限対処

- LocalStorageの上限は通常 **5MB**
- 保存時にサイズをチェックし、4MB超過で警告トースト、5MB超過で保存をブロックしてエラーを表示
- エラー時は「不要なプロジェクトを削除してください」と案内する
- IndexedDBへのフォールバックは将来拡張候補（§11）

### 10-1. 資料の英語化（第3タブ）

#### 概要

日本語マークダウン全体をAIでビジネス英語に英訳し、**新規プロジェクト**として保存する（元の日本語プロジェクトは変更しない）。左右対訳プレビューで翻訳内容を確認しながらHTML/PPTXダウンロードまでを行う。

> **翻訳スタイル指針**: 翻訳結果は**ビジネス英語**（Business English）で表現する。ビジネスシーンで使用される正式かつ簡潔な表現を優先し、カジュアルな口語表現や和製英語は使用しない。

#### 動作フロー

1. 資料の作成タブで開いているプロジェクトを資料の英語化タブが**自動引き継ぎ**
2. 「英語に翻訳して保存」ボタンをクリック
3. 確認ダイアログを表示：「選択中のスライドを英訳し、新規プロジェクトとして保存します。よろしいですか？」
4. 確認後、マークダウン全文をAzure OpenAIに送信し英訳を依頼
5. 翻訳中はボタンをローディング状態にし、全操作を無効化
6. 翻訳完了後、新規プロジェクトとして保存：
   - プロジェクト名: `{元のプロジェクト名} (EN)`
   - マークダウン: 翻訳後の全文（Marpフロントマターはそのまま保持）
7. 完了トースト「英語版を「{プロジェクト名} (EN)」として保存しました」を表示
8. 右ペインの英語版プレビューを更新し、HTML/PPTXダウンロードボタンを活性化

> **トークン上限対処**: 翻訳送信前にマークダウンのトークン数を推定し、128kの80%（約102k tokens）を超える場合は「スライド枚数が多すぎるため翻訳できません。スライドを分割してください」の警告ダイアログを表示して送信を中止する

#### スライドサムネイル同期

- 下部サムネイルをクリックすると左右のMarpプレビューが同じスライドページに同期する

#### 翻訳プロンプト（System Prompt）

```
1. Marpフロントマター（---で囲まれたYAMLブロック）は変更しない
```

### 10-2. スクリプト作成（第4タブ）

#### 概要

日本語スクリプトと英語スクリプトを左右対訳で表示・編集できる専用タブ。スライドサムネイルと連動してスクロールし、発表の準備を効率化する。

#### 動作フロー

**STEP 1 — 日本語スクリプト生成**

1. 資料の作成タブで開いているプロジェクトをスクリプト作成タブが**自動引き継ぎ**
2. 「日本語スクリプトを生成」ボタンをクリック
3. 日本語マークダウン全文をAzure OpenAIに送信
4. 生成中はボタンをローディング状態にし、操作を無効化
5. 生成完了後、スクリプトをスライドごとに区切って左ペインに表示（**編集可能なテキストエリア**）
6. 生成済みスクリプトをLocalStorageにキャッシュ（キー: `slidecraft:script:jp:<uuid>`）。キャッシュには生成時点のプロジェクト `updatedAt` と `jpScriptUpdatedAt`（日本語スクリプト最終更新時刻）を保存する

**STEP 2 — 英語スクリプト生成**

1. 「英語に翻訳」ボタンをクリック（日本語スクリプト未生成の場合はグレーアウト）
2. **左ペインの日本語スクリプト全文**をAzure OpenAIに送信し英訳を依頼
3. 生成中はボタンをローディング状態にし、操作を無効化
4. 生成完了後、英語スクリプトをスライドごとに右ペインに表示（**読み取り専用**）
5. 生成済み英語スクリプトをLocalStorageにキャッシュ（キー: `slidecraft:script:en:<uuid>`）。キャッシュには生成時点の `sourceJpScriptUpdatedAt`（日本語スクリプトの `jpScriptUpdatedAt`）を保存する

**キャッシュ無効化**
- **日本語スクリプトキャッシュ**: プロジェクトの `updatedAt` が変わった場合に無効化し、再生成ボタンを表示する
- **英語スクリプトキャッシュ**: `EnScriptCache.sourceJpScriptUpdatedAt` ≠ `JpScriptCache.jpScriptUpdatedAt` の場合に無効化する（日本語スクリプトを手動編集した場合も確実に検知）
- トークン上限超過時: 送信前にトークン数を推定し128kの80%を超える場合は警告ダイアログを表示して送信を中止する

#### スライドサムネイル同期

- 下部サムネイルをクリックすると左右両ペインが該当スライドのスクリプト位置に同期スクロールする

## 📐 12. スライド作成ルール

### 基本ルール

- **マークダウンの内容を忠実に反映する** — 勝手に内容を追加・削除しない
- **1セクション = 1スライド**が基本（内容が多い場合は2枚に分割してよい）

---

### チェックシート準拠ルール（必須）

以下は社内プレゼン作成チェックシートに基づく必須ルール。

#### 1. 目的を押さえる

- **ぼやかした表現を使わない**: 「〇〇を検討する」「〇〇を推進する」など主語・期限・責任者が不明な表現は禁止
- **次の行動が明確**: クロージングスライドで「誰が・いつまでに・何をするか」を明示する

#### 2. ストーリー・表現

- **全スライドにキーメッセージ/結論を必須配置**: スライドヘッダー直下に「このスライドで何が言いたいか」を1文で記載する（配置場所を全スライドで統一）
- **体言止め・主語なし名詞表現はNG**: キーメッセージは必ず主語＋述語の完全文で書く
  - NG例: 「データ品質の課題」
  - OK例: 「データ品質の低さがAI活用の最大の阻害要因である」
- **1スライド1メッセージ**: 1枚のスライドで伝えることは1つに絞る
- **大枠→細部の順**: いきなり細部から入らず、まず全体像を提示する
- **資料冒頭にサマリを記載**: タイトル直後にエグゼクティブサマリスライドを入れる

#### 3. デザイン

- **フォント**: 日本語は `游ゴシック`（既定）、英語は `Calibri` を優先
- **本文フォントサイズは `text-base`（16px）以上**: `text-sm`（14px）は注釈・補足のみ使用可
- **Suntoryロゴを全スライドのフッター右側に必須配置**: Marpカスタムテーマの CSS `section::after` で固定配置する（AIによる毎回挿入は誤削除リスクがあるため禁止）
  - 使用ファイル: `public/suntory-logo.png`（リポジトリに含める前に法務・広報部門の使用許諾を必ず確認すること）
  - 実装例: `styles/marp-theme.css` の `section::after { content: url('/suntory-logo.png'); position: absolute; bottom: 16px; right: 24px; width: 80px; }`
- **デザイン4原則を意識**:
  - **近接**: 意味のある塊をグルーピング、余白を有効活用
  - **整列**: 文字/オブジェクトの端を揃える（中央揃えとバラバラ配置は避ける）
  - **反復**: フォント・サイズ・色・スライド構成を全体で統一（スライドマスタ相当）
  - **コントラスト**: 強調したい要素にのみコントラストを使う
- **文字強調は色ではなく太さ・大きさで表現する**: `<strong>` タグを活用し、色による強調は避ける
- **ページ番号必須**: ヘッダー右端に `NN / MM` 形式で表示（既定テンプレートで実装済み）
- **スライドには絵文字を使用しない**: 絵文字はチャット返答（§13）にのみ使用する。スライドコンテンツでは代わりにアイコン画像（Iconify）を使う
- **文中に改行コードを挿入しない**: スライド幅に収まる文章に `<br>` などの明示的な改行は入れない（表示崩れの原因）

---

### スライド構成の設計

パターンに応じたスライド構成を生成する。必ず含めるスライド：

1. **タイトルスライド** — タイトル・日付・作成者の会社名、部署名、氏名（キーメッセージも1行で）
2. **アジェンダスライド** — 全セクションの目次
3. **各セクションスライド** — マークダウンのセクション順に生成
4. **まとめスライド** — ゴール・次のステップ（ある場合）

#### スライドの内容生成ルール

- セクションタイトルはそのままスライドタイトルに使う
- 箇条書きは `<ul>/<li>` または `<ul class="two-column">` で整形
- 重要ポイント（`note`フィールドや★マーク）は `<div class="highlight-box">` で強調

#### キーメッセージバーの生成（全セクションスライドに必須）

スライドヘッダーの直下に、必ずキーメッセージバーを挿入する。Marp内ではTailwindCSSは動作しないため、**インラインスタイルのみ**使用する：

```html
<!-- キーメッセージ（必須・体言止めNG・主語＋述語で1文） -->
<div style="padding: 8px 20px; background: #F5F5F5; border-bottom: 1px solid rgba(84,195,225,0.3); flex-shrink: 0;">
  <p style="font-size: 14px; font-weight: bold; color: #1a3a5c; line-height: 1.4; margin: 0;">
    ▌ {このスライドの結論を主語＋述語の完全文1文で。体言止め・ぼやかした表現は禁止}
  </p>
</div>
```

キーメッセージの書き方ルール：

| 種別 | 例 |
|---|---|
| NG（体言止め） | 「データ品質の課題」「AI活用の推進」 |
| NG（ぼやかし） | 「データ品質を検討する」「AI活用を推進する」 |
| OK | 「データ品質の低さが全社AI活用の最大の阻害要因である」 |
| OK | 「Option Aを採用し2027年Q4までに全社展開を完了する計画を提案する」 |

## 📝 13. マークダウン出力スタイルガイド（チャット返答専用）

> このセクションは **AIのチャット返答（ストーリー設計の会話メッセージ）** のスタイルガイドである。スライドコンテンツ（マークダウンのスライド本文）には適用しない。スライドのルールは §12 を参照。

AIがチャット返答として出力するテキストは、人間が読んで直感的に理解できるよう、以下の方針に従って作成する。

### ✅ 基本方針

- **絵文字を積極的に使う**: 見出し・箇条書きの先頭に絵文字を付けて視覚的に分かりやすくする
- **内容に合った絵文字を選ぶ**: 課題なら ⚠️、ゴールなら 🎯、コストなら 💰、スケジュールなら 📅 など
- **読み流せる構造にする**: 重要度に応じて `**太字**` や `> 引用` を使い、パッと見て要点が伝わるようにする
- **過剰な絵文字は避ける**: 1つの箇条書きに絵文字は1つまで。文中への多用はしない

### 🖋️ 絵文字の使い方例

```markdown
## 🎯 本日のゴール
- ✅ 投資可否の承認
- 📋 方針の合意
- ⚠️ 検討に戻すポイントの確認

## 💰 コスト試算
| 区分 | 金額 |
|---|---|
| 🏗️ 初期費用 | 500万円 |
| 🔄 年間運用費 | 120万円 |

## ⚠️ リスクと対策
- **リスク**: ベンダーロックイン
- **対策**: マルチクラウド構成を採用
```

### エラー種別と対処方針

| エラー種別 | トリガー | UI表示 | リトライ |
|---|---|---|---|

| Marpレンダリングエラー | スライドプレビュー更新 | プレビューエリアに「レンダリングエラー: マークダウンの構文を確認してください」 | 自動（マークダウン編集で再トリガー） |
| Iconify API 障害 | アイコン検索 | 検索結果欄に「アイコン検索が利用できません。URLを直接入力してください」 | 手動リトライボタン |
| 認証エラー（Cookie失効） | 全画面 | ログイン画面にリダイレクト | 再ログイン |
| ファイルインポート: 非対応形式 | `.md` 以外のファイル選択 | トースト「`.md` ファイルのみインポートできます」 | なし（再選択） |
| ファイルインポート: 読み取りエラー | ファイル読み取り失敗 | トースト「ファイルの読み取りに失敗しました。ファイルを確認してください」 | なし（再選択） |
| ファイルインポート: Marpフロントマターなし | `.md` にフロントマターが未付与 | 自動でMarpフロントマター（テーマ・フォント設定）を先頭に付与して読み込む（ユーザー通知なし） | — |
| トークン上限超過 | AI全機能（翻訳・スクリプト生成・編集チャット） | 「スライド枚数が多すぎます。スライドを分割するか不要なスライドを削除してください」警告ダイアログ | なし（手動対応） |

- すべての変数はブラウザのDevToolsから参照できてはならない
- OpenAI SDKでのAzure接続は `AzureOpenAI` クラスを使用する（`openai` npmパッケージ同梱）：

```typescript
import { AzureOpenAI } from 'openai';

const client = new AzureOpenAI({
  apiKey: process.env.AZURE_OPENAI_API_KEY,
  endpoint: process.env.AZURE_OPENAI_ENDPOINT,
  deployment: process.env.AZURE_OPENAI_DEPLOYMENT,
  apiVersion: '2024-02-01',
});
```

### Content Security Policy（CSP）

Next.js の `next.config.js` に以下のCSPヘッダーを設定し、XSSリスクを軽減する：

```js
// next.config.js
headers: [
  {
    source: '/(.*)',
    headers: [
      { key: 'Content-Security-Policy',
        value: "default-src 'self'; script-src 'self'; img-src 'self' https://api.iconify.design data:; style-src 'self' 'unsafe-inline'; connect-src 'self' https://api.iconify.design;" },
      { key: 'X-Frame-Options', value: 'DENY' },
      { key: 'X-Content-Type-Options', value: 'nosniff' },
    ],
  },
]
```

### LocalStorageのセキュリティリスク

- 社内機密情報がLocalStorageに平文保存されることを利用者に周知する
- CSPによりXSSリスクを軽減するが、完全な保護は不可能
- 機密度の高いプレゼン内容はブラウザ共有PCで作成しないよう運用ルールを設ける

```typescript
  updatedAt: string;             // ISO 8601形式
  parentProjectId?: string;      // 英語版プロジェクトの場合に元プロジェクトのUUIDを設定。日本語版はundefined
}

// 日本語スクリプトキャッシュ（キー: "slidecraft:script:jp:<uuid>"）
interface JpScriptCache {
  projectId: string;          // 対象プロジェクトのUUID
  sourceUpdatedAt: string;    // スクリプト生成時点のプロジェクト updatedAt（無効化判定用）
  jpScriptUpdatedAt: string;  // 日本語スクリプト最終更新時刻（手動編集時に更新。ISO 8601形式）
  script: string;             // 生成済み日本語スクリプト全文（"スライド N:\n..." 形式）
}

// 英語スクリプトキャッシュ（キー: "slidecraft:script:en:<uuid>"）
interface EnScriptCache {
  projectId: string;               // 対象プロジェクトのUUID
  sourceJpScriptUpdatedAt: string; // 翻訳時点の JpScriptCache.jpScriptUpdatedAt（無効化判定用）
  script: string;                  // 生成済み英語スクリプト全文（"Slide N:\n..." 形式）
}
```
